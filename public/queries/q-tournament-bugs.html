Query: Tournament Bugs — Diagnosis and Fixes

Root cause analysis and fix recommendations for the three failure modes observed in a 16-player bot tournament.

Date answered: 2026-04-11
Last updated: 2026-04-11 (all three bugs fixed — commit e6c9c06b)
Sources: 2026-04-11-ur-codebase
Related: tournament-flow, nakama-runtime, matchmaking

Observed Symptoms

From a 16-player tournament run:

  1. Some players stuck in waiting room at tournament start — didn’t get routed to a match until they refreshed the app
  2. Late time forfeits — players who had to refresh eventually got hit with AFK forfeits later in the game, even after successfully reconnecting and playing multiple turns
  3. General AFK/forfeit timer questions — whether the timer is accurate and responsive, and whether players are effectively ejected from the tournament

Bug 1 — Waiting Room at Tournament Start

✅ FIXED — commit e6c9c06b. Option A implemented: push notification on bracket creation.

Root Cause

A polling gap between bracket creation and the client seeing it.

When the 16th player joins, maybeStartBracketForRun is called synchronously. It calls createSingleEliminationBracket, then writes the bracket via updateRunWithRetry (an OCC write loop). This write takes a non-zero amount of time. Any client that polls get_public_tournament before the write commits sees no bracket in storage, causing buildPublicParticipationState to return state: "lobby", canLaunch: false.

The client with this stale response shows the waiting room UI and does not retry until the next normal poll interval. Players who refreshed forced a fresh RPC call that saw the committed bracket state.

Fix Implemented

Option A — Push notification on bracket creation (shared/tournamentNotifications.ts + backend/modules/tournaments/public.ts):

After maybeStartBracketForRun successfully writes the bracket, sendBracketReadyNotifications fires a Nakama push notification to all registered (non-bot) users. Notification code: 41_001, subject: tournament_bracket_ready.

Idempotency: a bracketStartNotificationToken (random UUID-style string) is written into the run’s metadata during the same OCC write that creates the bracket. Notifications are only sent if the token in the committed record matches the token generated by this RPC invocation — so concurrent maybeStartBracketForRun calls can’t each send a full blast of notifications.

The client (src/tournaments/useTournamentDetail.ts) attaches to socket.onnotification while the tournament detail screen is mounted. On receipt of a tournament_bracket_ready notification matching the current runId, it immediately calls refresh() — bypassing the 4-second poll interval entirely. Falls back gracefully to polling if the socket is unavailable.

Bug 2 — Race Condition on Match Launch

✅ FIXED — commit e6c9c06b. nk.matchCreate moved inside updateRunWithRetry callback.

Root Cause

nk.matchCreate was called before the idempotency guard write, creating a window where two separate matches could be created for the same bracket entry.

Both players assigned to a bracket entry call rpcLaunchTournamentMatch at roughly the same time. The old code:

Execution timeline:

Player A: reads entry → matchId: null → nk.matchCreate → match MA
Player B: reads entry → matchId: null → nk.matchCreate → match MB  ← orphaned
Player A: updateRunWithRetry → writes matchId: MA → succeeds
Player B: updateRunWithRetry → retries → sees matchId: MA → skips → returns MA to B

Result: Both players eventually join MA (correct). MB is an orphaned Nakama match with zero presences that times out.

Fix Implemented

nk.matchCreate moved inside the updateRunWithRetry callback, behind the idempotency guard:

One match per entry, guaranteed. Also added an explicit error throw if resolvedMatchId is null (was previously silently returning undefined).

Bug 3 — Late Forfeit After Reconnect

✅ FIXED — commit e6c9c06b. Fix A implemented: resetAfkOnMeaningfulAction called on rejoin in matchJoin.

Root Cause

accumulatedMs in PlayerAfkState was a running total across the entire match lifetime and was never reset on reconnect. A player who disconnected and reconnected within the grace window had that offline time counted toward their AFK total. Over multiple slow turns, the total crossed ONLINE_AFK_FORFEIT_MS (30,000ms) and triggered a forfeit — even for a player who was actively present.

The Scenario

  1. Player refreshes during a match → matchLeave fires → disconnectedAt set
  2. Player reconnects within 15s grace window → matchJoin fires → disconnect cleared, timer resumed
  3. Problem: accumulatedMs was not reset. The 8-12 seconds they were disconnected still counted.
  4. Over subsequent turns where they take 8+ seconds each, accumulatedMs crosses 30,000ms
  5. Forfeit fires despite the player being actively engaged

Fix Implemented

In matchJoin (backend/modules/index.ts), when processing presences joining an active match, resetAfkOnMeaningfulAction(state, playerColor, nowMs) is now called if:

  • The player had no prior presence before this join (!hadPresenceBeforeJoin), or
  • Joining cleared a disconnect grace (didClearDisconnectGrace)

This resets accumulatedMs and timeoutCount to 0 and stamps lastMeaningfulActionAtMs = nowMs. The disconnect time no longer counts toward the AFK total.

Two edge cases are handled correctly:

  • True rejoin after disconnect: AFK debt clears ✅
  • Concurrent second session joins without disconnect gap (e.g., player opens a second tab while first is still open): AFK debt is preserved ✅ (neither !hadPresenceBeforeJoin nor didClearDisconnectGrace is true in this case)

New test coverage in authoritativeTimer.rpc.test.ts verifies both cases.

Timer Accuracy

The timers run at TICK_RATE = 10 (every 100ms). This means:

  • Timer resolution is ±100ms — not perfectly precise but more than adequate for 10s turns
  • No drift accumulation: each tick uses Date.now() for disconnect checks rather than counting ticks, so wall-clock accuracy is maintained
  • The AFK accumulator uses tick-counting (+= 100ms per tick) which can drift slightly if ticks are delayed (e.g., server load). This is unlikely to matter in practice.

Tournament Ejection After Forfeit

When a player is forfeited (either via turn timer, AFK accumulator, or disconnect grace expiry):

  • The match ends with the other player as winner
  • completeTournamentBracketMatch runs: winner advances, forfeited player → state: "eliminated"
  • The forfeited player is removed from further tournament participation

This is working correctly — the ejection itself is not buggy. The bugs above are about forfeits triggering when they shouldn’t (Bug 3) or matches not starting correctly (Bugs 1 & 2).
