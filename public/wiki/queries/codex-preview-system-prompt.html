Codex Prompt: Preview System (Phase 3)

Saved prompt for delegating the cosmetic preview system implementation to Codex.

Last updated: 2026-04-15 Related: economy-overview, cosmetic-store, layer-frontend, game-engine, zustand-game-store Depends on: codex-cosmetic-store-prompt (Phase 2 must be complete first)

Prompt

You are implementing the cosmetic preview system for the Royal Game of Ur — a multiplayer mobile/web game built with Expo (React Native) + Nakama backend. The repo is at https://github.com/micho8cho93/ur.

This is Phase 3 of the economy build. Phases 1 (currency wallet) and 2 (cosmetic data model + basic store) are already complete. The store screen at app/(game)/store.tsx exists and has “Preview” buttons that currently show a “Preview coming soon” toast. Your job is to replace those with a working live preview system.

The preview system is the most important conversion lever in the store — players buy what they can see working. This is not a nice-to-have; it is the core of Phase 3.

What already exists (from Phase 2)

  • shared/cosmetics.ts — CosmeticDefinition, OwnedCosmetic, StorefrontResponse types
  • backend/modules/cosmeticCatalog.ts — static cosmetic catalog, getCosmeticById(id)
  • src/store/StoreProvider.tsx — useStore() hook with storefront, purchaseItem
  • app/(game)/store.tsx — Featured + Daily Rotation grid; “Preview” button on each card shows a toast placeholder
  • Soft currency wallet is live; players have balances

Do not modify the backend in this phase. All work is frontend-only.

Codebase context

Frontend stack: Expo SDK (React Native), Expo Router (file-based routing at app/), Zustand (store/useGameStore.ts), NativeWind (Tailwind utility classes for styling), React Context for feature providers.

Game engine (logic/): Pure TypeScript, no React or Nakama dependencies.

  • createInitialState(matchConfig?) — builds a fresh GameState (all pieces at position -1, phase: 'rolling')
  • applyMove(state, move) — returns a new GameState (immutable, deep clone)
  • getValidMoves(state, roll) — returns legal MoveAction[]
  • rollDice() — returns 0–4
  • Core types: GameState, PlayerColor, Piece, Player, GamePhase, MoveAction

Board rendering (in components/): The existing game screen (app/(game)/game.tsx) renders the board using a hierarchy:

  • GameBoard.tsx — container component; receives gameState and renders the board grid
  • Tile.tsx — individual tile; receives tile type (normal | rosette | war), any pieces on that tile, and styling
  • Piece.tsx — individual piece; receives owner color (light | dark) and position state

Before writing any code, read components/GameBoard.tsx, components/Tile.tsx, and components/Piece.tsx to understand their current props and styling approach. The instructions below assume you have done this. The cosmetic theme system you design must integrate with however these components currently style themselves — do not assume; verify first.

Zustand store (store/useGameStore.ts): The game store holds the live match GameState. You will NOT use the game store for the preview — the preview uses its own isolated local GameState managed in React component state, completely separate from any live match.

What to build

1. Cosmetic theme system (shared/cosmeticTheme.ts + src/store/CosmeticThemeContext.tsx)

Goal: Allow the board rendering components (GameBoard, Tile, Piece) to visually reflect a cosmetic without threading cosmetic props through every layer. The preview wraps them in a theme provider; the real game screen uses defaults (no theme provider needed).

Important: The defaults must exactly reproduce the current visual appearance. Nothing should look different in the real game screen after this change.

src/store/CosmeticThemeContext.tsx — React context:

When useCosmeticTheme() is called outside a CosmeticThemeProvider, it returns the resolved defaults (safe fallback, no error thrown).

2. Update board components to consume the theme

Modify Tile.tsx and Piece.tsx to read their colors from useCosmeticTheme() instead of hardcoded values. The only change is the color source — layout, sizing, shadows, borders, and all other styling remain exactly as they were.

Specifically:

  • Tile.tsx: replace hardcoded tile background colors with theme.board.normalTileColor, .rosetteTileColor, .warTileColor depending on tile type; replace border color with theme.board.borderColor
  • Piece.tsx: replace hardcoded piece fill with theme.pieces.lightPrimaryColor / theme.pieces.darkPrimaryColor depending on owner; replace border with the matching lightBorderColor / darkBorderColor
  • If there is a dice rendering component, apply theme.dice there as well
  • GameBoard.tsx: replace any hardcoded board background with theme.board.backgroundColor

Regression requirement: The real game screen (app/(game)/game.tsx) must look pixel-identical after this change. Write a quick visual regression checklist in a comment at the top of each modified component listing the original hardcoded values, so reviewers can verify nothing shifted.

Do NOT add CosmeticThemeProvider to the real game screen or the app provider stack. The real game always uses defaults. Only the preview modal wraps in a provider.

3. Cosmetic → theme mapping (shared/cosmeticTheme.ts)

This is where cosmetic assetKey values are mapped to actual colors. For the MVP, use a color mapping table — no image assets yet. Define the mapping directly in this file:

cosmeticDefinitionToTheme looks up the id in COSMETIC_COLOR_MAP. If not found, returns {} (no override — defaults apply).

4. Preview game state factory (logic/previewState.ts)

The preview needs a visually interesting static board — not an empty board, not a real match in progress. Create a pure function that returns a hand-crafted GameState:

The preview state should show:

  • Light player: 2 pieces in the shared war zone (path indices 4–11), 1 piece near the end of its private lane, 1 piece still in reserve (position -1)
  • Dark player: 2 pieces in the war zone (on different tiles than light), 1 piece near its finish, 1 piece in reserve
  • phase: 'rolling' (the rolling phase is the stable “waiting” state — no valid moves shown)
  • currentTurn: 'light'
  • rollValue: null

Use createInitialState() as the base, then mutate a deep clone to position the pieces. Do not call applyMove — just directly set piece.position values on the cloned state. Reference logic/pathVariants.ts and logic/constants.ts to pick positions that will visually distribute pieces across both the private lanes and the war zone.

This state is used for all cosmetic types (boards, pieces) — it gives context regardless of what’s being previewed.

5. Preview modal component (components/CosmeticPreviewModal.tsx)

A full-screen modal (or large bottom sheet) that opens when a user taps “Preview” on any store card.

Props:

Layout (top to bottom):

  1. Header bar: Close button (X, top-left), cosmetic name (center, bold), tier badge (top-right, color-coded). Height: ~56px.

  2. Preview board area (dominant, ~60% of screen height):

    • Renders a GameBoard wrapped in CosmeticThemeProvider with cosmeticDefinitionToTheme(cosmetic) as the theme
    • Uses createPreviewGameState() as the game state — never connects to a live match
    • For board skins and piece sets: show the static previewGameState with theme applied
    • For dice animations: show a simplified dice face display (not a full board). Render the dice component with theme.dice applied. Include a “Roll” button that re-rolls a random 0–4 value and shows it (client-side Math.random() only, no RPC)
    • For emotes: show a simple animation placeholder (“Emote preview coming soon” with a visual indicator that this type’s preview is in a later phase)
    • The board renders at its natural aspect ratio, centered, with padding

  3. Comparison row (if relatedCosmetics has items):

    • Horizontal scrollable row of small thumbnail chips (colored rectangle + name)
    • The currently previewed cosmetic is highlighted
    • Tapping a chip switches the previewed cosmetic instantly (updates the CosmeticThemeProvider theme, no reload)
    • Maximum 5 items shown (clip the rest)
    • Label: “Compare with others in this section”

  4. Info row: Tier badge + tier label (“Legendary”, “Epic”, etc.) + price (currency icon + amount). If owned: show “Owned” badge instead of price.

  5. Action button (bottom, full width):

    • If not owned: “Buy for [price] [currency icon]” — primary button, calls onBuy(cosmetic)
    • If owned: “Equipped” / “Owned” — disabled secondary button style
    • 56px height, large tap target

State the modal manages locally:

  • activeCosmeticId: string — which cosmetic is currently shown (starts as props.cosmetic.id, updates when comparison chip tapped)
  • diceRollValue: number — for dice animation preview only (random 0–4, re-rolled on button tap)
  • Derive activeCosmetic = [cosmetic, ...relatedCosmetics].find(c => c.id === activeCosmeticId)
  • Derive activeTheme = cosmeticDefinitionToTheme(activeCosmetic)

Performance: The GameBoard inside the modal is rendered once on open. Theme changes (comparison chip taps) only update context value — React.memo on Tile and Piece must still work correctly (their memo comparators compare primitive color values from theme, not object references). Verify this doesn’t cause full re-renders of all tiles on theme switch.

Styling: bg-slate-950 for modal background, consistent with the store screen dark theme. Use Modal from React Native (transparent: true, animationType: 'slide').

6. Wire the preview into the store screen (app/(game)/store.tsx)

Replace the “Preview coming soon” toast with the real modal:

  • Add state: previewCosmetic: CosmeticDefinition | null and previewVisible: boolean
  • On “Preview” button tap: set previewCosmetic to the tapped item, set previewVisible: true
  • Pass relatedCosmetics as the other items from the same section (daily rotation items if tapped from daily, featured items if from featured)
  • onBuy handler: close the preview modal first, then call purchaseItem(cosmetic.id) from useStore(), handle success/error with toast — same flow as the direct “Buy” button on cards
  • onClose: set previewVisible: false, clear previewCosmetic after modal animation completes (100ms delay to avoid visual flash)

What NOT to build in this phase

  • Continuous game simulation loop (pieces auto-moving, dice auto-rolling on a timer) — deferred to Phase 3+. The preview is a static board with an optional manual dice roll for dice-type cosmetics.
  • 3D board rotation (pinch-to-zoom, gyroscope) — deferred.
  • Emote preview animation — placeholder only. Full emote animation system is separate.
  • Server-side cosmetic preview rendering — all preview logic is client-side.
  • Asset pipeline (actual image/texture files per cosmetic) — color-only MVP. Real asset loading comes when art is ready.
  • The preview_item RPC — the frontend already has all cosmetic data from get_storefront; no backend call needed for preview.

Acceptance criteria

  • shared/cosmeticTheme.ts exports CosmeticTheme, BoardTheme, PiecesTheme, DiceTheme, resolveBoardTheme, resolvePiecesTheme, resolveDiceTheme, cosmeticDefinitionToTheme
  • src/store/CosmeticThemeContext.tsx exports CosmeticThemeProvider and useCosmeticTheme(); calling useCosmeticTheme() outside a provider returns resolved defaults without error
  • Tile.tsx and Piece.tsx read colors from useCosmeticTheme() with no visual change to the real game screen (regression comment block present)
  • logic/previewState.ts exports createPreviewGameState() returning a valid GameState with pieces distributed across both lanes and the war zone
  • COSMETIC_COLOR_MAP in shared/cosmeticTheme.ts has entries for all cosmetics in the Phase 2 catalog
  • CosmeticPreviewModal renders the board with the correct cosmetic theme applied
  • Comparison chip row allows switching between related cosmetics; theme updates instantly without re-mounting the board
  • Dice cosmetics show a dice face with a “Roll” button; tapping it shows a new random value
  • Emote cosmetics show a placeholder message
  • “Buy” button in the modal calls onBuy correctly; “Owned” state is reflected
  • Preview modal opens from every “Preview” button in the store screen (Featured and Daily Rotation sections)
  • tsc --noEmit passes with no new errors
  • Real game screen (app/(game)/game.tsx) looks pixel-identical before and after (verify against regression comment blocks in Tile.tsx and Piece.tsx)
