Codex Prompt: Store Navigation + Admin Control Panel

Saved prompt for delegating store navigation and admin panel implementation to Codex.

Last updated: 2026-04-16 Related: economy-overview, cosmetic-store, ur-internals, layer-frontend, layer-backend Depends on: Phases 1–3 complete (currency wallet, cosmetic data model + basic store, preview system)

Prompt

You are extending the Royal Game of Ur — a multiplayer mobile/web game built with Expo (React Native) on the frontend and Nakama (self-hosted, TypeScript runtime) on the backend. The monorepo layout is:

ur/
├── app/              ← Expo (React Native) player-facing app
├── components/       ← shared React Native UI components
├── src/              ← feature providers and contexts
├── store/            ← Zustand store
├── services/         ← NakamaService singleton
├── logic/            ← pure game engine (no React, no Nakama)
├── shared/           ← client-server contract types
├── backend/modules/  ← Nakama TypeScript runtime
└── ur-internals/     ← separate admin web app (React, web-only)

Phases 1–3 are complete:

  • Soft currency wallet live (Nakama wallet API, soft_currency key)
  • shared/cosmetics.ts — CosmeticDefinition, OwnedCosmetic, StorefrontResponse types
  • backend/modules/cosmeticCatalog.ts — static in-memory catalog (12–15 cosmetics), getCatalog(), getCosmeticById()
  • backend/modules/storeRotation.ts — getDailyRotation(), getFeaturedItems()
  • RPCs: get_storefront, purchase_item, get_owned_cosmetics
  • src/store/StoreProvider.tsx — useStore() hook
  • app/(game)/store.tsx — Featured tab + Daily Rotation grid (MVP; full tab system deferred)
  • shared/cosmeticTheme.ts — CosmeticTheme, cosmeticDefinitionToTheme()
  • src/store/CosmeticThemeContext.tsx — board theme context
  • components/CosmeticPreviewModal.tsx — preview modal with comparison cycling

This task has two independent workstreams. Do them in order.

Workstream A — Store entry point + full category tabs (player-facing app)

A1. Store button on the lobby screen

Read app/(game)/lobby.tsx before writing anything. Understand the existing layout and navigation patterns before adding to it.

Add a Store button to the lobby screen. Placement: wherever it fits naturally alongside existing navigation elements (match queue button, tournament button, profile link, etc.) — use your judgment after reading the screen. Don’t disrupt the existing layout hierarchy.

The button should:

  • Display a cart or market icon (use whatever icon library is already imported in the project) + “Store” label
  • Navigate to app/(game)/store.tsx using Expo Router’s router.push('/store') or the routing pattern already established in this screen
  • Show the player’s current soft currency balance as a small badge or subtitle on the button (read from useStore())
  • Follow the existing button style conventions in the lobby — do not introduce new design patterns

A2. Full category tab system in the store screen

The existing app/(game)/store.tsx only has Featured + Daily Rotation. Add the full tab system.

Read the existing store screen fully before touching it. Understand how it is structured, then extend it.

Add a tab bar at the top (or wherever it fits the existing layout) with these tabs:

TabContents
FeaturedExisting Featured + Daily Rotation sections (no change — this is the current screen)
BoardsAll board-type cosmetics from the catalog
PiecesAll piece set cosmetics
DiceAll dice animation cosmetics
EmotesAll emote cosmetics

Tab implementation notes:

  • Use a scrollable horizontal tab bar if the screen is already using a scroll view; otherwise use a sticky tab bar at the top
  • Each category tab shows ALL cosmetics of that type from the full catalog — not just what’s in the daily rotation. This is a browsing view, not a rotation view.
  • Items in category tabs are sorted: unowned first (sorted by tier: Legendary → Epic → Rare → Common), owned items at the bottom with an “Owned” badge
  • Each item card in category tabs has the same design as the daily rotation cards (same component, same behavior — tapping “Preview” opens the preview modal, “Buy” triggers the purchase flow)
  • Add a small “In rotation today” indicator tag on items that happen to be in the current daily rotation — a subtle visual cue, not a separate filter
  • The get_storefront RPC already returns ownedIds and the daily rotation — use this data. For the full catalog browse, call a new get_full_catalog RPC (defined in Workstream B, section B2, but implement the frontend call here and handle the loading state)

Tab state: the selected tab is local component state, not in the Zustand store or URL. It resets to “Featured” each time the screen mounts.

Workstream B — Admin control panel (ur-internals)

Before starting: read the existing ur-internals codebase

Read ur-internals/src/ fully before writing any code. Understand:

  • What framework and component library is used (the wiki doesn’t specify; discover it from the code)
  • The existing auth flow (ur-internals/src/auth/nakama.ts) and session handling
  • The RPC client pattern (ur-internals/src/api/client.ts, unwrapRpcPayload, tryParseJsonString)
  • What pages/routes already exist and their navigation structure

The admin panel pages you add must match the existing UI framework, routing, styling, and RPC call patterns exactly. Do not introduce a new UI library.

B1. Critical prerequisite: migrate the catalog from static code to Nakama storage

This is the most important architectural change in this task. The current getCatalog() returns a hardcoded in-memory array. The admin panel needs to read and write the catalog at runtime. Without this migration, nothing in the admin panel can persist.

Migration approach:

  1. Create a new Nakama storage collection for the catalog:

    CollectionKeyUserIdContents
    cosmetics_catalogitems'system'{ items: CosmeticDefinition[], version: number, updatedAt: string }

  2. Add a backend function loadCatalogFromStorage(nk): Promise<CosmeticDefinition[]> in backend/modules/cosmeticCatalog.ts that:

    • Reads cosmetics_catalog/items/system from Nakama storage
    • If the record doesn’t exist (first boot), seeds it from the existing hardcoded array and writes it to storage, then returns it
    • If it exists, returns the items array
    • Caches in module-level memory with a 60-second TTL (in-process cache to avoid repeated storage reads on every storefront request). Cache is invalidated when admin writes occur.

  3. Update getCatalog() to call loadCatalogFromStorage. Since this changes it from sync to async, update all callers:

    • get_storefront RPC — already async, just await the catalog load
    • purchase_item RPC — same
    • get_owned_cosmetics RPC — same
    • storeRotation.ts functions — update signatures to accept catalog as a param (caller loads it)

  4. Add invalidateCatalogCache() — a module-level function that clears the in-memory TTL cache. Call this after any admin write to the catalog.

  5. The original hardcoded array stays in the file as SEED_CATALOG: CosmeticDefinition[] — the seed data, used only when the storage record doesn’t exist yet.

Do not break existing behavior: after this migration, get_storefront, purchase_item, and get_owned_cosmetics must behave identically from the player’s perspective.

B2. New backend RPCs (admin-only)

All new admin RPCs must:

  • Be registered in backend/modules/index.ts via initializer.registerRpc(...)
  • Call requireAdminRole(ctx, nk) at the top — this is the existing admin auth check used in tournament admin RPCs (read backend/modules/tournaments/auth.ts to find the exact function name and call pattern)
  • Wrap payloads with the existing parseRpcPayload / tryParseJsonString pattern
  • Return JSON.stringify(result) as the RPC return value

admin_get_full_catalog (no payload):

  • Calls loadCatalogFromStorage(nk)
  • Returns { items: CosmeticDefinition[] }
  • Also called by the player-facing get_full_catalog RPC (see below)

get_full_catalog (no payload, player-callable — NOT admin-only):

  • Same as above but no admin role check — any authenticated player can call this to browse all cosmetics
  • Returns { items: CosmeticDefinition[] }
  • Uses the same loadCatalogFromStorage with caching

admin_upsert_cosmetic (payload: CosmeticDefinition):

  • Validates the payload shape matches CosmeticDefinition (required fields: id, name, tier, type, price, rotationPools, rarityWeight, releasedDate, assetKey)
  • If id already exists: updates that item in the catalog array (merge — only overwrite provided fields)
  • If id doesn’t exist: appends a new item
  • Writes the updated catalog back to cosmetics_catalog/items/system using OCC
  • Calls invalidateCatalogCache()
  • Also invalidates the store rotation cache (call invalidateRotationCache() if it exists in storeRotation.ts, otherwise add it)
  • Returns { success: true, item: CosmeticDefinition }

admin_disable_cosmetic (payload: { id: string }):

  • Soft-delete: adds disabled: true to the cosmetic definition (update CosmeticDefinition in shared/cosmetics.ts to include disabled?: boolean)
  • Disabled cosmetics are filtered out by loadCatalogFromStorage before returning to callers (players never see them; admin panel shows them with a “Disabled” badge)
  • Calls invalidateCatalogCache(), invalidateRotationCache()
  • Returns { success: true }

admin_enable_cosmetic (payload: { id: string }):

  • Reverses admin_disable_cosmetic: sets disabled: false
  • Same invalidation, same return shape

admin_get_rotation_state (no payload):

  • Reads the current rotation record from store_state/rotation/system (created in Phase 2)
  • Returns { dailyRotationIds: string[], featuredIds: string[], limitedTimeEvents: LimitedTimeEvent[], generatedAt: string, manualOverride: boolean }

admin_set_manual_rotation (payload: { dailyRotationIds?: string[], featuredIds?: string[] }):

  • Overwrites the store_state/rotation/system record with the provided IDs
  • Sets a manualOverride: true flag on the record so the auto-algorithm doesn’t regenerate it on the next get_storefront call
  • Validates that all provided IDs exist in the catalog and are not disabled
  • Returns { success: true, rotation: { dailyRotationIds, featuredIds } }

admin_clear_manual_rotation (no payload):

  • Sets manualOverride: false on the rotation record and clears dailyRotationIds / featuredIds
  • The next get_storefront call will regenerate the rotation via the normal algorithm
  • Returns { success: true }

admin_set_limited_time_event (payload: LimitedTimeEvent):

  • Upserts the event into the limitedTimeEvents array on the rotation record
  • Active events are included in StorefrontResponse.limitedTime when get_storefront is called
  • Returns { success: true, event: LimitedTimeEvent }

admin_remove_limited_time_event (payload: { id: string }):

  • Removes the event with matching id from limitedTimeEvents
  • Returns { success: true }

admin_get_store_stats (no payload):

  • Reads the analytics collection to summarize purchase data
  • Returns: { totalPurchases: number, purchasesByCosmetic: Record<string, number>, totalSoftCurrencySpent: number }
  • If analytics data is sparse or missing, return zeros — don’t throw

Update get_storefront to respect the manualOverride flag: if manualOverride: true, use the stored dailyRotationIds and featuredIds directly instead of running getDailyRotation().

Update get_storefront to include active limited-time events in StorefrontResponse.limitedTime based on startsAt/endsAt window vs. current server time.

B3. Admin panel pages in ur-internals

Add the following pages to ur-internals. Match the existing routing, navigation, and UI framework exactly.

Page 1: Catalog Manager

Route: wherever new routes are added (read the existing router to find the pattern)

Purpose: View, add, edit, and disable cosmetics in the catalog.

Layout:

  • Header: “Catalog Manager” title + “Add Cosmetic” button (top-right)
  • Filter bar: dropdowns for tier (all | common | rare | epic | legendary) and type (all | board | pieces | dice | emote); toggle for “Show disabled”
  • Cosmetic table: one row per cosmetic with columns:
    • Tier badge (color-coded)
    • Name
    • Type
    • Price (amount + currency type)
    • Rotation pools (comma-separated tags: “daily”, “featured”, “limited”)
    • Rarity weight
    • Status (“Active” | “Disabled”)
    • Actions: “Edit” button, “Disable” / “Enable” toggle button
  • Add / Edit form (modal or inline panel):
    • Fields for all CosmeticDefinition properties: id (readonly on edit), name, tier (dropdown), type (dropdown), price.amount, price.currency (dropdown: soft | premium), rotationPools (multi-select checkboxes), rarityWeight (0.0–1.0 number input), assetKey, releasedDate, availabilityWindow.start (optional), availabilityWindow.end (optional)
    • “Save” calls admin_upsert_cosmetic; “Cancel” closes without saving
    • Validation: id must be non-empty, kebab-case; price.amount must be positive integer; rarityWeight must be 0.0–1.0
    • On save success: close form, reload catalog table
    • On error: show inline error message (do not close form)

Data: Load via admin_get_full_catalog on mount. Show loading state while fetching.

Page 2: Rotation Control

Route: separate from Catalog Manager

Purpose: Inspect and control what’s in the store rotation.

Layout (three sections):

Section 1 — Current Rotation

  • Shows the current daily rotation and featured items (from admin_get_rotation_state)
  • Each item shows: name, tier badge, type, asset key
  • If manualOverride: true: shows a yellow ”⚠ Manual override active” banner
  • If manualOverride: false: shows “Auto-generated” with the generatedAt timestamp and time until next reset
  • “Clear Override” button (visible when manualOverride: true): calls admin_clear_manual_rotation, reloads

Section 2 — Manual Override

  • Two multi-select dropdowns (or drag-and-drop lists) populated from the full catalog:
    • “Daily Rotation (max 8)”: select up to 8 cosmetics from common/rare pool
    • “Featured Items (max 2)”: select up to 2 cosmetics from epic/legendary pool
  • “Apply Override” button: calls admin_set_manual_rotation with selected IDs; shows success/error feedback
  • Validation: daily rotation must have at least 6 items; featured must have at least 1 if set

Section 3 — Limited-Time Events

  • Table of existing events with columns: Name, Cosmetics (count), Starts, Ends, Status (Active / Scheduled / Expired)
  • “Add Event” button opens a form:
    • Fields: id, name, cosmeticIds (multi-select from catalog), startsAt (datetime picker), endsAt (datetime picker)
    • “Save” calls admin_set_limited_time_event
  • “Remove” button per row calls admin_remove_limited_time_event after a confirmation dialog

Data: Load via admin_get_rotation_state and admin_get_full_catalog on mount.

Page 3: Store Stats (simple)

Route: separate page

Purpose: Quick overview of purchase activity.

Layout:

  • Summary cards: Total purchases, Total soft currency spent (from admin_get_store_stats)
  • Table: one row per cosmetic that has been purchased, showing name and purchase count, sorted by count descending
  • “Refresh” button to reload
  • No date filtering needed at this stage — all-time totals only

Navigation

Add links to all three new pages in the existing ur-internals navigation (sidebar, nav bar, or wherever nav links live — read the existing structure). Labels: “Catalog”, “Rotation”, “Store Stats”.

Security note on admin auth

The existing admin auth in ur-internals has a known critical vulnerability (BUG-A01: hardcoded custom ID allows privilege escalation). Do not fix or change the auth system in this task — that is a separate security task. Use the existing auth flow exactly as-is. Your admin RPCs on the backend must call requireAdminRole(ctx, nk) at the top of each handler — this is already the correct server-side enforcement mechanism regardless of the client-side auth vulnerability.

What NOT to build in this task

  • Premium currency IAP flow
  • Battle pass / seasonal pass
  • Analytics dashboards beyond the simple store stats page
  • Cosmetic asset upload (images, animations) — assetKey is a string reference; actual assets are managed outside this system for now
  • Fixing BUG-A01 (admin auth vulnerability) — separate task
  • Per-player purchase history view

Acceptance criteria

Workstream A:

  • Store button visible on lobby screen; tapping it navigates to the store
  • Lobby store button shows current soft currency balance
  • Store screen has a working tab bar: Featured | Boards | Pieces | Dice | Emotes
  • Category tabs show all cosmetics of that type from the full catalog, sorted (unowned by tier, owned at bottom)
  • “In rotation today” tag shown on items that appear in current daily rotation
  • Preview modal opens correctly from category tab cards (same behavior as Featured tab)
  • Purchase flow works from category tabs

Workstream B — Catalog migration:

  • loadCatalogFromStorage(nk) seeds from SEED_CATALOG on first run; reads from storage on subsequent calls
  • 60-second in-process cache with invalidateCatalogCache() function
  • All existing RPCs (get_storefront, purchase_item, get_owned_cosmetics) work identically from the player’s perspective after the migration
  • CosmeticDefinition has optional disabled?: boolean field in shared/cosmetics.ts

Workstream B — Admin RPCs:

  • All admin RPCs call requireAdminRole and reject non-admin callers
  • get_full_catalog (player-callable) returns full non-disabled catalog
  • admin_upsert_cosmetic adds new and updates existing cosmetics; persists to storage; invalidates cache
  • admin_disable_cosmetic / admin_enable_cosmetic toggle visibility; disabled cosmetics hidden from players
  • admin_set_manual_rotation / admin_clear_manual_rotation control the manualOverride flag; get_storefront respects it
  • admin_set_limited_time_event / admin_remove_limited_time_event manage events; get_storefront includes active events
  • admin_get_store_stats returns purchase totals without throwing when data is sparse

Workstream B — Admin UI:

  • Catalog Manager loads and displays all cosmetics including disabled ones (when toggle on)
  • Add/Edit form validates inputs and shows inline errors; saves via admin_upsert_cosmetic
  • Disable/Enable toggle calls correct RPCs and reflects new state immediately
  • Rotation Control shows current rotation with manual override status
  • Manual override form validates item counts and calls admin_set_manual_rotation
  • Clear Override button resets to auto-generated rotation
  • Limited-time event table shows all events; Add/Remove work correctly with confirmation on remove
  • Store Stats page shows purchase totals and per-cosmetic table
  • All three pages linked in the existing ur-internals navigation
  • tsc --noEmit passes with no new errors across all packages
