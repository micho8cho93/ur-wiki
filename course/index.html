Module 02 — The Cast

The shared/ folder — a contract between two worlds

Code in shared/ runs on both the app and the server. It defines the language they use to talk to each other.

CODE 
export const MatchOpCode = {
  ROLL_REQUEST:  1,
  MOVE_REQUEST:  2,
  STATE_SNAPSHOT: 100,
  SERVER_ERROR:   101,
} as const;
PLAIN ENGLISH

A dictionary of message types that client and server share…

Code 1 = "I want to roll the dice"

Code 2 = "I want to make this move"

Code 100 = "here is the full game state" (server → clients)

Code 101 = "something went wrong on the server"

"as const" = these numbers can never be changed accidentally

One definition, two users. If the app sends a move with code 2 and the server expects code 2, they stay in sync forever. Change it in one place and both sides update instantly.

Module 03 — The Engine

Rolling the dice — 4 binary coins

The ancient game used 4 pyramid-shaped dice, each a coin flip. Result: 0 to 4. The code is elegant and testable.

CODE — logic/engine.ts 
export const rollDice = (
  randomSource = Math.random
): number => {
  let sum = 0;
  for (let i = 0; i < 4; i++) {
    if (randomSource() >= 0.5) sum++;
  }
  return sum;
};
PLAIN ENGLISH

Create "rollDice" — it accepts an optional random source…

(Default: Math.random. Tests can pass a fake one for predictable results.)

Start counting at zero.

Flip each of the 4 "coins" one by one…

If the flip is ≥ 0.5, count it as 1 (marked side up).

Return the total — 0 (all blank) to 4 (all marked).

Dependency injection. randomSource = Math.random means tests can swap in a controlled function so the dice always roll a specific value. This is how you write code that's both real and testable — a core skill to ask AI for.
Module 03 — The Engine

Calculating valid moves — the bouncer's checklist

After rolling, the engine checks every piece against a checklist. Only pieces that pass every check become valid moves.

1

Roll of zero?

No moves possible. Return empty list immediately — turn is lost.

2

Already finished?

Skip pieces that have already crossed the finish line.

3

Would land past the end?

Must roll the exact number to bear off — overshooting is illegal.

4

Own piece already there?

Can't stack — skip this move.

5

Opponent on the war rosette?

Pieces on the center rosette are invulnerable — can't capture them.

✓

Valid move — add it to the list

All checks passed. This piece can move.

Module 03 — The Engine

Applying a move — the photocopier pattern

applyMove() never changes the existing state. It makes a perfect copy, modifies the copy, and returns it. The original is always preserved.

CODE — logic/engine.ts 
export const applyMove = (
  state: GameState,
  move: MoveAction
): GameState => {
  const newState = JSON.parse(
    JSON.stringify(state)
  ) as GameState;
  // all changes go on newState only
  return newState;
};
PLAIN ENGLISH

A function that takes current game state + a move…

…and returns a new game state (same shape).

Deep-clone the entire state by serializing to text…

…then immediately parsing back to an object. Totally independent copy.

All edits happen on newState — original "state" is untouched.

Return the new, modified state.

Immutability = never changing data in place, always creating a new copy. This makes it trivial to "undo" a move, replay match history, or debug — the old state is always intact.

Module 04 — The Server

The neutral third party neither player can fool.

Nakama is an open-source game server that handles matchmaking, real-time messaging, leaderboards, and player storage — so you don't have to build any of it yourself.

🏛️

What is Nakama?

A ready-made backend designed for games. Think of it as a managed game server platform — auth, real-time, storage, all included.

⚖️

Why a server at all?

Without a server, each player's device could claim whatever moves it wanted. The server is the authoritative source of truth.

Server-authoritative means the server runs the game engine itself. It doesn't trust what players say happened — it calculates what actually happened.

❌ Client-authoritative

Each device decides what happened. A cheater modifies their app to always win. No defense.

The server runs applyMove() itself. It decides what's real. Neither player can lie to it.

Module 05 — The Full Picture

Zustand — why one global scoreboard beats passing notes

React apps have a tree of components. Getting data from a grandparent to a grandchild means passing it through every level. Zustand breaks that chain — any component reads directly from the store.

CODE — any component 
const roll = useGameStore(
  (state) => state.roll
);
const gameState = useGameStore(
  (state) => state.gameState
);
PLAIN ENGLISH

Subscribe to just the "roll" function from the global store…

(This component re-renders ONLY when "roll" changes — nothing else.)

Subscribe to just the "gameState" object…

(Only re-renders when gameState changes — efficient and predictable.)

Selective subscriptions = performance. The store has 50+ fields. A dice component only subscribes to the roll value. The board doesn't flicker when the timer ticks. Each component only re-renders when its specific data changes.
Module 05 — The Full Picture

When things go wrong — resilience patterns

Production apps need to handle network drops, timing edge cases, and disconnections gracefully.

🔄

Reconnection grace window

Lose connection mid-game? The server holds your spot. Reconnect in time and the game continues as if nothing happened.

⏱️

AFK timer

If a player goes idle too long, the game auto-forfeits. Prevents abandoned games from locking out opponents.

🔁

Silent session refresh

Auth tokens expire. The Nakama service refreshes them in the background using a refresh token — you never get logged out mid-match.

🤝

Rematch flow

After a game, both players can accept a rematch. If either declines or times out, the request resolves gracefully.
