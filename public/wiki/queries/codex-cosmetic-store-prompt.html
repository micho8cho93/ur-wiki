Codex Prompt: Cosmetic Data Model + Basic Store

Saved prompt for delegating the cosmetic store implementation to Codex.

Last updated: 2026-04-15 Related: economy-overview, cosmetic-store, monetization, progression-currency, layer-backend, layer-frontend, nakama-runtime

Prompt

You are implementing the cosmetic data model and basic store for the Royal Game of Ur — a multiplayer mobile/web game built with Expo (React Native) on the frontend and Nakama (self-hosted, TypeScript runtime) on the backend. The repo is at https://github.com/micho8cho93/ur.

Codebase context

Backend (backend/modules/): TypeScript modules that run inside Nakama’s sandbox. All backend logic follows a few established patterns you must respect:

  • Storage uses Nakama’s 3-tier key { collection, key, userId } with optimistic concurrency control (OCC). Writes retry up to 3 times on version conflict. Existing collections: user_profiles, progression, challenges, matches, analytics.
  • RPCs are registered in backend/modules/index.ts via initializer.registerRpc(...). Each handler receives (ctx, logger, nk, payload: string) and must manually JSON.parse(payload) and JSON.stringify the return value. Existing RPCs include private_match_create, private_match_code_verify, list_spectatable_matches, rank_match_queue, get_progression.
  • Match end hook (finalizeCompletedMatch → awardWinnerProgression) is where XP and challenge grants currently fire. Soft currency grants should be added here in the same pattern as XP.
  • Nakama wallet API: nk.walletsUpdate([{ userId, changeset: { soft_currency: +15 }, metadata: {} }]) — use this for currency grants and deductions. Do not roll a custom ledger; use native Nakama wallets.
  • Shared types live in shared/ and are imported by both frontend and backend. Add new shared types here.

Frontend (app/, src/, store/, services/): Expo SDK with Expo Router (file-based routing), Zustand for state (store/useGameStore.ts), NativeWind for styling (Tailwind utility classes), and a NakamaService singleton at services/nakama.ts for RPC calls and WebSocket.

The provider stack in app/_layout.tsx is:

AuthProvider → EloRatingProvider → ProgressionProvider → ChallengesProvider → ScreenTransitionProvider → [screens]

New feature providers follow this same pattern.

Screens live under app/(game)/ (authenticated). A new store screen goes at app/(game)/store.tsx.

2. Cosmetic catalog (backend/modules/cosmeticCatalog.ts)

Seed an initial catalog of 12–15 cosmetics as a static in-memory array (no database seeding needed at launch; the catalog is static server-side code). Cover at minimum:

  • 4–5 board skins: 2 common (soft 300–500), 2 rare (soft 500–800 or low premium), 1 epic (premium 800–1200)
  • 4–5 piece sets: same tier distribution
  • 2–3 dice animations: 1 common (soft 400), 1 rare (soft 600), 1 epic (premium 900)
  • 1–2 emotes: common (soft 200)

Name them with ancient Mesopotamian flavor: Lapis, Cedar, Obsidian, Gold, Alabaster, Carnelian, Ivory, Bronze, etc.

Include a getCatalog(): CosmeticDefinition[] export and a getCosmeticById(id: string): CosmeticDefinition | undefined helper.

3. Rotation algorithm (backend/modules/storeRotation.ts)

Implement getDailyRotation(catalog, today: string, previousRotations: string[][]): CosmeticDefinition[] that:

  1. Filters to items in the 'daily' rotation pool whose availability window includes today
  2. Runs a weighted random sample of 8 items (use rarityWeight as the weight)
  3. Ensures type diversity: ≥1 board, ≥1 piece set, ≥1 animation/emote. If a type is missing after sampling, replace the lowest-weight selected item of another type with a random item of the missing type.
  4. Applies anti-duplication: excludes items that appeared in previousRotations[0] (yesterday). Items in previousRotations[1] or previousRotations[2] are allowed but deprioritized (halve their weight before sampling).

For getFeaturedItems(catalog): CosmeticDefinition[]: return the 1–2 Epic/Legendary items with the highest rarityWeight whose availability window is active.

4. Nakama storage collections

Add two new collections:

CollectionKeyUserIdContents
cosmeticsownedplayer’s userId{ items: OwnedCosmetic[] }
store_staterotation'system' (global){ dailyRotation: string[], featuredIds: string[], generatedAt: string, previousDays: string[][] }

The rotation record is global (not per-user). It is regenerated once per 24h by the get_storefront RPC if generatedAt is more than 24h old. Use OCC on all writes.

5. Backend RPCs

Register all three in backend/modules/index.ts:

get_storefront (no payload required):

  1. Read global rotation from store_state/rotation/system. If missing or stale (>24h), generate a new daily rotation using getDailyRotation, write it back to storage.
  2. Read the calling player’s cosmetics/owned/{userId} to get their owned set.
  3. Return a StorefrontResponse (defined in shared types).
  4. Cache TTL: the RPC itself is not cached, but the rotation record in storage is the cache (regenerated at most once per 24h).

purchase_item (payload: { itemId: string }):

  1. Look up itemId in the catalog. If not found, throw 'ITEM_NOT_FOUND'.
  2. Read the player’s cosmetics/owned from storage. If they already own this item, throw 'ALREADY_OWNED'.
  3. Determine price (soft or premium currency) from the cosmetic definition.
  4. Call nk.walletsUpdate to deduct the currency. If the wallet has insufficient funds, Nakama will throw — catch it and return 'INSUFFICIENT_FUNDS'.
  5. Append the new OwnedCosmetic to their owned list and write back to storage with OCC.
  6. Return { success: true, cosmeticId: itemId, updatedWallet: { soft_currency: number, premium_currency: number } }.

get_owned_cosmetics (no payload):

  1. Read cosmetics/owned/{userId} from storage.
  2. Return { items: OwnedCosmetic[], cosmeticIds: string[] }.

6. Frontend: StoreProvider (src/store/StoreProvider.tsx)

Following the same pattern as ProgressionProvider:

  • On mount (authenticated), call the get_storefront RPC via NakamaService and store the result in local React state.
  • Expose via context hook useStore(): { storefront: StorefrontResponse | null, loading: boolean, purchaseItem: (itemId: string) => Promise<PurchaseResult>, refresh: () => void }.
  • purchaseItem calls the purchase_item RPC, updates the ownedIds set in local state on success, and propagates the error string on failure.
  • Add StoreProvider to the provider stack in app/_layout.tsx, inside ChallengesProvider.

7. Frontend: Store screen (app/(game)/store.tsx)

Build the MVP store screen — Featured tab and Daily Rotation grid only. The full tab system (Boards, Pieces, Dice, Bundles) comes later.

Layout (top to bottom):

  1. Header: “Store” title + wallet balance display (soft currency amount, coin icon). Read from useStore() or Nakama wallet. Update after each purchase.

  2. Featured section (if storefront.featured.length > 0):

    • Horizontal scrollable row of large cards (one per featured cosmetic)
    • Each card: cosmetic name, tier badge (color-coded: Common=gray, Rare=blue, Epic=purple, Legendary=gold), price, “Preview” button (stub for now — shows a toast “Preview coming soon”), “Buy” button
    • Section title: “Featured”

  3. Daily Rotation section:

    • Section title: “Daily — refreshes in Xh Ym” (countdown to storefront.rotationExpiresAt)
    • 2-column grid of cosmetic cards
    • Each card: cosmetic thumbnail (use a placeholder image tied to assetKey), name, tier badge, price, “Buy” button
    • Cards for already-owned items show an “Owned” badge and disable the Buy button

  4. Purchase flow:

    • Tapping “Buy” opens a simple confirmation bottom sheet: cosmetic name, price, “Confirm” and “Cancel” buttons
    • On confirm: call purchaseItem, show a loading indicator, then either a success toast (“You own [name]!”) or an error toast (“Not enough coins” / “Purchase failed”)
    • 2-tap maximum to purchase (card → confirm sheet → done)

  5. Empty / loading state: spinner while storefront is null; error message if the RPC fails.

Styling: NativeWind (Tailwind). Use bg-slate-900 for the screen background, match the app’s existing visual language (see other (game)/ screens for reference).

No real assets yet: Use a solid-color rectangle as the cosmetic thumbnail, color-coded by tier (gray/blue/purple/gold). A real asset pipeline comes in a later phase.

Soft currency earning (wire-in to match end)

In backend/modules/index.ts, inside finalizeCompletedMatch → awardWinnerProgression, after XP is granted:

  • Award 15 soft currency to the winner and 5 soft currency to the loser using nk.walletsUpdate.
  • This mirrors exactly how XP is awarded. Add it as a parallel call, not a replacement.
  • Log the wallet update: logger.info('Soft currency awarded', { userId, amount, matchId }).
  • The grant should be idempotent: store a flag in the match result object (softCurrencyAwarded: true) so retries skip the wallet update.

What NOT to build in this task

  • The preview system (live board with cosmetic applied) — deferred
  • Premium currency IAP flow (Apple/Google billing) — deferred
  • The full tab system (Boards, Pieces, Dice tabs) — deferred
  • Personalized rotation (user affinity tracking) — deferred
  • Bundles — deferred
  • Any cosmetic visual assets beyond colored placeholder rectangles

Acceptance criteria

  • shared/cosmetics.ts exports CosmeticDefinition, OwnedCosmetic, StorefrontResponse, BundleDefinition types
  • backend/modules/cosmeticCatalog.ts has ≥12 cosmetics covering all 4 types and at least 3 tiers
  • getDailyRotation returns exactly 8 items with type diversity and anti-duplication
  • get_storefront RPC returns a valid StorefrontResponse; rotation regenerates after 24h; owned IDs are included
  • purchase_item RPC deducts currency, grants cosmetic, rejects double-purchase and insufficient funds
  • get_owned_cosmetics RPC returns the player’s cosmetic collection
  • Match end grants 15 soft currency to winner and 5 to loser via Nakama wallet API; grant is idempotent
  • StoreProvider loads storefront on mount, exposes useStore() hook, handles purchase
  • Store screen renders Featured + Daily Rotation, shows Owned badge, confirms purchase in ≤2 taps, shows countdown to next rotation reset
  • No TypeScript errors (tsc --noEmit passes)
  • All new backend RPCs have at least basic unit tests (happy path + error cases for purchase_item)
